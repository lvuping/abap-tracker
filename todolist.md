# SY-UNAME 추적기 개선 계획 및 현재 과제

이 문서는 SAP ABAP SY-UNAME 추적기의 완성도를 높이기 위한 구체적인 실행 계획과 현재 파악된 한계점을 정리한 것입니다.

---

## 🚀 완성도 높은 추적기를 위한 4단계 개선 계획

현재의 강력한 기반 위에, 추적기의 정확성, 분석 범위, 사용성을 극대화하기 위한 4단계 개선 계획을 제안합니다.

### 1단계: 코어 엔진 강화 - 더 깊고 정확한 추적
- **목표**: 추적의 정확도를 높이고, 복잡한 로직 흐름을 더 잘 이해하도록 코어 로직을 개선합니다.
- **주요 과제**:
    - [ ] **`PERFORM` 파라미터 추적**: `USING`, `CHANGING` 파라미터를 분석하여 서브루틴으로 전달되는 변수 흐름을 추적합니다.
    - [ ] **객체지향(OO) ABAP 분석 심화**: `METHOD` 호출의 파라미터(`EXPORTING`, `IMPORTING` 등)와 클래스 속성(Attribute)의 오염 상태를 추적합니다.
    - [ ] **제어구조 분석 정교화**: `LOOP` 내부에서의 변수 변경, `IF` 분기문에 따른 오염 전파 경로를 더 정교하게 분석하여 오탐을 줄입니다.

### 2단계: 패턴 확장 - 최신 ABAP 및 엣지 케이스 지원
- **목표**: 더 많은 ABAP 구문을 인식하여 분석 커버리지를 높입니다.
- **주요 과제**:
    - [ ] **최신 ABAP (7.4+) 구문 지원**: `DATA(...)` 인라인 선언, `NEW #(...)` 생성자, 테이블 표현식 등 최신 구문 패턴을 추가합니다.
    - [ ] **동적(Dynamic) 구문 탐지 강화**: `PERFORM (lv_form)`, `UPDATE (lv_table)` 등 동적 구문을 '분석 불가능한 위험 지점'으로 명확히 식별하고 경고합니다.
    - [ ] **필드 심볼(`FIELD-SYMBOLS`) 추적 고도화**: `ASSIGN COMPONENT` 등 동적 필드 접근을 더 정교하게 추적합니다.

### 3단계: 결과 리포팅 개선 - 실행 가능한 인사이트 제공
- **목표**: 분석 결과를 단순 정보 나열이 아닌, 사용자가 쉽게 이해하고 조치할 수 있는 형태로 가공합니다.
- **주요 과제**:
    - [ ] **신뢰도 점수(Confidence Score) 도입**: 모든 결과에 '높음/중간/낮음' 신뢰도를 부여하여 결과의 신뢰 수준을 알려줍니다.
    - [ ] **추적 경로 시각화**: 추적 경로를 그래프나 다이어그램 형태로 시각화하는 기능을 구상합니다. (e.g., Graphviz 연동)
    - [ ] **보안 및 품질 힌트 제공**: DB 테이블의 키 필드에 `SY-UNAME`이 사용되는 경우 등 잠재적 보안 취약점에 대한 경고를 추가합니다.

### 4단계: 사용성 향상 - 개발 워크플로우에 통합
- **목표**: 개발자가 더 쉽고 편리하게 추적기를 사용할 수 있도록 개선합니다.
- **주요 과제**:
    - [ ] **프로젝트 전체 분석**: 지정된 폴더 내 모든 ABAP 소스코드를 스캔하여 `SY-UNAME` 사용처를 자동으로 찾아 분석하는 기능을 구현합니다.
    - [ ] **설정 파일 도입**: 분석 깊이, 대상 테이블 패턴(`Z*`, `Y*`), 제외할 오브젝트 등을 `config.yaml` 같은 설정 파일로 관리하여 유연성을 높입니다.

## 🎯 현재 추적기의 한계점 및 개선 과제 (맹점 정리)

사용자 피드백과 추가 분석을 통해 파악된 현재 추적기의 한계점과 이를 해결하기 위한 구체적인 개선 과제는 다음과 같습니다.

1.  **정확한 분석 범위 제어**
    - **문제점**: 현재 분석기는 `SY-UNAME` 위치로부터 고정된 라인 수(e.g., 1000줄)를 분석 범위로 삼고 있어, 스코프가 끝나는 지점(`PERFORM`, `ENDFORM` 등)을 지나서도 불필요한 분석을 계속할 수 있습니다.
    - **개선 과제**: `PERFORM`, `METHOD` 호출, `ENDFORM`, `ENDMETHOD` 등 명확한 스코프 종료 구문을 만나면, 그 즉시 해당 분기의 추적을 멈추도록 로직을 수정해야 합니다. 예를 들어, 50줄 뒤에 `PERFORM`이 나온다면, 그 경로의 분석은 50줄에서 멈춰야 합니다.

2.  **명확한 '추적 실패' 상태 정의**
    - **문제점**: 추적 결과 Sink를 찾지 못했을 때, 그 이유가 명확하지 않을 수 있습니다.
    - **개선 과제**: 최종적으로 `Z/Y` 테이블 Sink를 찾지 못하면, 결과 상태에 "Not Found"를 명확히 표시하고, 그 이유를 구체적으로 명시해야 합니다. (e.g., "스코프 내에서 Z/Y 테이블 Sink를 찾지 못했습니다.")

3.  **내부 테이블(`APPEND`)을 통한 흐름 추적 강화**
    - **문제점**: `SY-UNAME`이 구조체(Work Area)에 할당되고, 이 구조체가 `APPEND` 구문을 통해 내부 테이블(Internal Table)에 추가된 후, 해당 내부 테이블이 `INSERT ... FROM TABLE` 이나 `MODIFY ... FROM TABLE` 구문으로 DB에 반영되는 복합적인 흐름을 놓칠 수 있습니다.
    - **개선 과제**: 내부 테이블을 '오염된 데이터의 컨테이너'로 인식하고 추적하는 기능이 필요합니다. 특정 내부 테이블이 오염되면, 해당 테이블을 사용하는 모든 DB 조작 구문(`INSERT`, `MODIFY` 등)을 Sink로 감지해야 합니다.

4.  **다양한 DB 조작(DML) 구문 패턴 지원**
    - **문제점**: `INSERT`, `MODIFY` 등은 매우 다양한 형태로 사용될 수 있습니다. (e.g., `INSERT (lv_table) FROM ...`, `MODIFY table FROM wa.`, `UPDATE ... SET (lt_set_clause)`)
    - **개선 과제**: `patterns.py`에 실제 사용 빈도가 높은 다양한 DML 구문 변형(특히 동적 구문 포함)을 지속적으로 추가하여 커버리지를 높여야 합니다.

5.  **최종 Sink로서의 Z/Y 테이블 명확화**
    - **문제점**: 추적의 최종 목표가 `Z` 또는 `Y`로 시작하는 커스텀 테이블이라는 점이 로직상 최우선 순위로 강조되지 않을 수 있습니다.
    - **개선 과제**: 분석 로직의 최종 목표는 '오염된 데이터가 Z/Y 테이블에 도달하는가'임을 명확히 하고, 다른 Sink(RFC, 감사 필드 등)는 부가적인 정보로 처리하거나 우선순위를 낮추도록 코드를 재확인하고 강화해야 합니다.

6.  **스코프 외부 사용 추론 기능**
    - **문제점**: 오염된 변수가 `PERFORM` 구문 직전에 마지막으로 나타나고 스코프 내에서 Sink를 찾지 못하는 경우, 추적이 그대로 종료되어 사용자는 이유를 알기 어렵습니다.
    - **개선 과제**: 이러한 경우, 결과에 "Scope Boundary Reached" 상태와 함께 "오염된 변수 'lv_variable'이(가) PERFORM 'subroutine' 외부에서 사용될 것으로 추측됩니다." 와 같은 구체적인 메시지를 추가하여 사용자가 다음 분석 단계를 예측할 수 있도록 도와야 합니다.

7.  **추가 고려사항 (Cline 자체 분석)**
    - **동적 구문 처리의 근본적 한계**: `ASSIGN COMPONENT (lv_field_name) ...` 처럼 변수로 필드명이 결정되는 경우, 정적 분석만으로는 어떤 필드가 오염되었는지 정확히 알 수 없습니다. 이런 구문은 '추적 불가능한 위험 지점'으로 분류하고 경고하는 것이 현실적인 대안입니다.
    - **변수 초기화(`CLEAR`) 미반영**: `CLEAR lv_variable.` 구문으로 오염된 변수가 초기화되면, 오염 상태가 제거되어야 하지만 현재 로직은 이를 반영하지 않습니다. 이는 오탐(False Positive)의 원인이 될 수 있습니다.
    - **선언부 컨텍스트 부족**: 분석기가 코드의 일부(snippet)만 보기 때문에, 변수의 정확한 타입(e.g., 구조체, 내부 테이블)을 알기 어렵습니다. 이는 구조체와 내부 테이블을 추론하는 데 한계로 작용합니다.
